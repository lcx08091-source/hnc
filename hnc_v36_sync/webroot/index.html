v3.4.12 LTS Critical Hotfix  2026-04-12
Critical 上传限速 隐藏 4 个版本
  • 🔥 修复 HNC 历史上最严重的隐藏 bug:上传限速在 Android 12+ / kernel 5.x+ 设备上完全失效。这个 bug 从 v3.4.1 开始存在,影响所有使用现代内核的设备(包括 RMX5010 SD8 Elite Android 16),持续了 11 个版本。修复来自真机上传限速失效的实测排查
  • 根因:Linux 4.5+ 内核引入了 clsact qdisc 作为老 ingress qdisc 的现代替代品,Android 12+ 默认在每个网络接口上预装 clsact(用于 BPF tether offload)。HNC 在 v3.4.1 加了 clsact 检测逻辑,检测到时跳过 tc qdisc add ingress。但是后续 tc filter add 仍然用了老 ingress 的语法 parent ffff:
  • 关键差异:clsact 提供两个独立的 hook handle — ffff:fff2(ingress 方向)和 ffff:fff3(egress 方向),必须显式指定。老 ingress qdisc 用 ffff: 裸 handle 默认就是 ingress。HNC 在 clsact 上用 parent ffff: 加 filter,filter 被挂到 egress hook 而不是 ingress,入站流量根本不经过这个 filter
  • 后果:wlan2 → ifb0 的 mirred 重定向 filter 没生效 → 设备的上传流量根本没进 ifb0 → ifb0 上的 HTB rate 限制毫无意义 → 上传方向走的是直连,完全不限速。但 ifb0 上的 class、u32 filter 都正常创建,流量统计字节数也正常更新(因为统计走 iptables HNC_STATS,跟 ifb0 无关),所以 user 完全看不出问题
  • 为什么之前没发现:(1) 错误信息被 2>/dev/null 吞掉,silent fail (2) WebUI 显示 toast 成功 + badge 显示 ↑X MB/s,user 以为生效了 (3) 下载限速完全正常(走 wlan2 root htb 不依赖 ifb),不容易意识到只有上传坏了 (4) 老 Android(kernel < 4.5)没 clsact,HNC 走老 ingress 路径语法是对的
  • 修复:init_tc 检测 clsact 时使用正确的 parent ffff:fff2,老 ingress 仍然用 parent ffff:。同时把 silent fail 改成 explicit log,以后类似问题立刻能从 logs/service.log 看到 ingress v4 mirred filter added on ffff:fff2 或 FAILED on ffff:fff2 (上传限速会失效!)
  • 验证方法(装上 v3.4.12 后):su; tc filter show dev wlan2 parent ffff:fff2 应该看到一行 filter ... mirred (Egress Redirect to device ifb0)。然后 speedtest 测上传应该被限到设定值附近
  • 影响范围:所有 v3.4.1 - v3.4.11 用户,只要你的设备 kernel 是 4.5+(Android 6+ 大部分都是),clsact 就会被预装,这个 bug 就触发。Android 12+ 必中招。Android 8-11 视 OEM 内核版本而定

    • clsact qdisc 的双 hook 设计:Linux 4.5 引入 clsact 是为了把 ingress 和 egress 的 filter 挂载点统一到一个 qdisc 上。它在 root 上挂一个 dummy parent ffff:fff1,内部分两个 minor handle:fff2 = ingress hook(对应老 ingress qdisc),fff3 = egress hook
    • 为什么 parent ffff: 在 clsact 上不是 ingress:tc 在 parent 没有 minor 时按 root 方向解释,clsact 上 root 方向是 egress,所以 filter 被挂到出站方向。设备的上行流量是从 wlan2 入站的,根本不会触发出站 filter
    • 诊断脚本输出 [4] 的解读:qdisc clsact ffff: parent ffff:fff1 后面是空的,正是因为 show 命令也用了老语法 parent ffff:,filter 实际被加到了 fff3 但 show 在 ffff: 上看不到 — 这是 100% 一致的因果链证据
    • 为什么 ifb0 上的 class + u32 filter 是对的:那一段代码独立于 ingress hook,不依赖 clsact。HNC 在 ifb0 自己的 root qdisc 上挂 HTB,然后用 u32 filter 按 src IP 分类,这一切都正常工作。问题只在"流量怎么进 ifb0"这一步
    • 为什么 ifb0 RX bytes 是 0:ifb0 接收流量唯一的方式是被 mirred action 重定向过去。mirred filter 没生效 → ifb0 RX 永远是 0 → class 1:N Sent 0 bytes → HTB rate 限制毫无意义 → 上传直连不限速
    • 修复后流量路径:wlan2 入包 → clsact ingress hook (fff2) → mirred filter 匹配所有(match u32 0 0) → redirect 到 ifb0 → ifb0 root htb → u32 src filter 按 IP 分类 → 1:N class (rate XMbit) → 限速生效 → 包从 ifb0 egress(loopback)→ 回到原 wlan2 ingress 路径 → 进入 conntrack/forward → 出 upstream 接口
    • silent fail 改 explicit log:之前 tc filter add ... 2>/dev/null || log "failed" 把 stderr 吞了再 log 一行通用消息。新版用 && log "added on $parent" 在成功路径明确 log 用了哪个 parent(便于反向追溯),失败时不吞 stderr
    • 已知不影响:延迟注入(netem)走的是 HTB class leaf qdisc,不依赖 ingress hook,延迟功能一直正常。下载限速走 wlan2 root htb,也不依赖 ingress hook。只有"上传限速 + 上传延迟"两个走 ifb0 路径的功能受影响 — v3.4.12 修了根因(filter parent),两者都恢复
